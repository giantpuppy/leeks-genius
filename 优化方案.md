# 排期助手 - 后续优化方向

## 方向一：Web 端数据持久化

### 价值

当前 Web 端数据存在内存中，刷新页面即丢失。实现持久化后，用户可在浏览器中长期使用，不依赖移动端。

### 实现路径

1. 引入 `shared_preferences` + `sqflite_common_ffi_web`（wasm 方案），或
2. 更轻量方案：用 `localStorage`/`IndexedDB` 实现 JSON 序列化存储，在 `database_helper_web.dart` 中读写
3. 优先推荐方案 2：在 Web DB 初始化后从 localStorage 加载，每次写操作后异步保存
4. 加一个简单的数据版本号，后续表结构变更时可自动迁移

### 验收路径
1. 在 Chrome 中添加一条剧目
2. 刷新页面，数据仍在
3. 关闭标签页重新打开，数据仍在
4. 清空 localStorage 后数据清空（符合预期）

## 方向二：Web 端图片上传与 OCR

### 价值
当前 OCR 功能依赖 `google_mlkit_text_recognition` 和 `image_picker`，两者均不支持 Web。实现 Web 端 OCR 后，PC 用户可直接上传票根截图识别排期。

### 实现路径

1. 图片上传：将 `image_picker` 替换为 `image_picker_for_web` 或 `file_picker`，在 Web 端走 `dart:html`/`dart:js_interop` 的文件选择 API
2. OCR 引擎：Web 端集成 `tesseract.js`（通过 `js` 包桥接），或调用后端 OCR API（如百度/腾讯/阿里云 OCR）
3. 移动端保持现有 `google_mlkit` 方案不变，用条件编译区分平台
4. 识别逻辑 `_parseOcrText` 保持复用

### 验收路径
1. 在 Chrome 中点击"识图"按钮
2. 选择一张票根截图（含日期、时间、剧场、票价信息）
3. 确认字段被正确提取并填充到表单中
4. 移动端验证原有 OCR 功能不受影响

---

## 方向三：深色模式 / 主题系统

### 价值
剧场环境通常光线较暗，深色模式是刚需。同时提升应用整体质感。

### 实现路径
1. 定义 `ThemeData` 的 dark 变体，使用 `ColorScheme.fromSeed(brightness: Brightness.dark)`
2. 将硬编码颜色替换为 `Theme.of(context).colorScheme.*` 取值
3. 添加设置项（Profile 页），用 `Provider`/`InheritedWidget` 或最简单的 `ValueNotifier` 管理主题状态
4. 跟随系统：监听 `MediaQuery.platformBrightnessOf(context)`

### 验收路径
1. 切换系统深色模式，应用自动跟随
2. 在设置页手动切换浅色/深色/跟随系统
3. 遍历所有页面，确认无硬编码白色/黑色导致的对比度问题
4. 截图对比，确认深色模式下所有文字可读、图标可见

## 方向四：数据导入导出（Excel/CSV）

### 价值
便于批量录入历史数据、与其他工具（Notion/Excel）互通、数据备份。

### 实现路径
1. 引入 `csv` 包实现 CSV 读写，`excel` 包实现 xlsx 读写
2. 导出：将 shows + performances + cast_members + actors 关联导出为结构化 Excel（多 sheet 或单 sheet 扁平化）
3. 导入：解析 CSV/Excel，按行创建数据库记录，支持重复数据检测（按剧目名+日期去重）
4. Web 端用 `file_picker` + Blob URL 下载，移动端用 `path_provider` + `share_plus`

### 验收路径
1. 导出：点击导出按钮，生成文件并下载，用 Excel 打开确认数据完整
2. 导入：准备一个包含 10 条记录的 CSV，导入后所有页面数据正确显示
3. 重复导入同一份文件，提示已存在或自动去重
4. 异常文件（空文件、格式错误）给出友好提示

## 方向五：演员追踪与卡司对比

### 价值
核心用户场景：追踪喜欢的演员排期、对比不同场次的卡司差异（决定是否加场）。

### 实现路径
1. 演员表增加 `isFavorite` 字段
2. Profile 页增加"我的关注"列表，展示关注演员的所有 upcoming 场次
3. 在 ShowDetail 页，同剧目不同场次之间做卡司对比（高亮差异角色）
4. 日历/甘特图增加筛选器：只看"有关注演员的场次"
5. 通知提醒：演出前 N 天提醒（移动端用 `flutter_local_notifications`，Web 用 `web_notifications`）

### 验收路径
1. 在演员列表标记几位演员为关注
2. 进入"我的关注"页面，确认只显示这些演员的场次，按时间排序
3. 打开一个有多个场次的剧目，对比两场卡司，差异角色被高亮
4. 筛选日历只显示关注演员的场次，确认过滤正确

---

## 方向六：统计看板升级

### 价值
当前统计较简单，升级后提供更有洞察的数据可视化，增强用户粘性。

### 实现路径
1. 用 `fl_chart` 扩展现有图表：
   - 月度观剧频次趋势（折线图）
   - 消费分布（饼图：按剧场 / 按剧目类型）
   - 演员出镜频次排行（横向柱状图）
2. 添加时间维度筛选：今年 / 去年 / 全部
3. 添加"观剧报告"功能：年度总结页，生成可分享的卡片（用 `RepaintBoundary` + `toImage` 导出图片）

### 验收路径
1. 添加多条不同月份、不同剧场、不同价位的记录
2. 切换统计维度，确认图表数据与数据库一致
3. 切换时间筛选器，确认范围过滤正确
4. 生成年度总结卡片，长按/右键保存图片，图片内容完整清晰

## 优先级建议

| 优先级 | 方向 | 理由 |
|--------|------|------|
| P0 | 方向一（Web 持久化） | 基础体验，刷新丢数据不可接受 |
| P0 | 方向三（深色模式） | 剧场场景刚需，实现成本低 |
| P1 | 方向二（Web OCR） | PC 端核心录入场景，但实现成本较高 |
| P1 | 方向五（演员追踪） | 核心用户价值，增强产品差异化 |
| P2 | 方向四（导入导出） | 数据互通，批量录入需求 |
| P2 | 方向六（统计升级） | 锦上添花，提升分享传播 |
